# 2.3 Создание серверной части с использованием Node.js

Серверная часть тестовой среды реализована на **Node.js** с использованием фреймворка **Express**. Приложение обеспечивает API для проверки работоспособности развёрнутой среды (статус сервисов, проверка связи), выдачу документации API и простую функциональность заметок для демонстрации клиент-серверного взаимодействия и прохождения unit-тестов в CI.

## Инициализация приложения

Была инициализирована серверная часть в виде модульного приложения: логика маршрутов и подключений вынесена в модуль `app.js`, а запуск HTTP-сервера выполняется в `server.js`. В `app.js` подключаются Express, клиенты PostgreSQL (`pg`) и Redis (`redis`), включается middleware для разбора JSON-тел запросов (`express.json()`). Сервер слушает порт из переменной окружения `PORT` (по умолчанию 8080). Исходный код доступен в репозитории проекта: `backend/app.js`, `backend/server.js`.

## Маршруты API

Реализованы следующие маршруты.

- **GET /api/status** — возвращает сводку о состоянии backend (время, hostname контейнера, NODE_ENV) и результат проверки подключения к PostgreSQL (тестовый запрос `SELECT NOW()`) и к Redis (команда PING). Параметры подключения к БД и Redis задаются переменными окружения (`POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `REDIS_URL`), что позволяет использовать один и тот же код в Docker Compose и в Kubernetes.

- **GET /api/hello** — возвращает приветственное сообщение и текущее время в формате JSON; используется для быстрой проверки доступности API с клиента.

- **GET /api/notes** и **POST /api/notes** — получение списка заметок и добавление новой заметки. Данные заметок хранятся в памяти процесса (массив в `app.js`). При создании заметки проверяется наличие и длина поля `text` (не более 200 символов); при ошибке валидации возвращается ответ с кодом 400. Эндпоинты используются для демонстрации взаимодействия клиента с сервером и покрываются unit-тестами (Jest, Supertest) в CI.

- **GET /api/api-docs** и **GET /api-docs.json** — возвращают OpenAPI 3.0 спецификацию (объект в памяти приложения) с описанием перечисленных эндпоинтов. Спецификация используется страницей «Документация API» на клиенте для отображения списка методов и путей.

Отдельные модули для авторизации (OAuth2, JWT) и CRUD стримов в данной работе не реализованы: серверная часть ориентирована на демонстрацию тестовой среды и её инфраструктуры, а не на бизнес-логику видеоплатформы.

## Работа с базой данных и Redis

Подключение к PostgreSQL выполняется с помощью клиента **pg** (модуль `Client`). Конфигурация (хост, порт, пользователь, пароль, имя базы) формируется из переменных окружения и по умолчанию указывает на сервис `db` в сети контейнеров. В рамках эндпоинта `/api/status` выполняется синхронный по смыслу запрос: установка соединения, выполнение `SELECT NOW()`, закрытие соединения; результат отображается в сводке статуса. Отдельные репозитории (например, StreamRepository, UserRepository) и ORM (SQLAlchemy) не используются: для текущих задач достаточно прямых запросов и проверки доступности БД.

Подключение к Redis выполняется через клиент **redis**. В эндпоинте статуса создаётся клиент по URL из переменной окружения (`REDIS_URL`), выполняется команда PING, соединение закрывается; результат (успех или текст ошибки) включается в ответ `/api/status`. Redis не используется для хранения сессий или токенов — только для проверки доступности сервиса в составе тестовой среды.

Таким образом, серверная часть не реализует асинхронные модели данных и сложные репозитории; она обеспечивает минимально необходимую логику для проверки развёртывания и демонстрации API.

## Dockerfile сервера

Для запуска серверной части в контейнере используется Dockerfile в каталоге `backend/`. В образе используется базовый образ **node:20-alpine**. В рабочую директорию копируются `package.json` и при наличии `package-lock.json`; затем выполняется `RUN npm install`, так что все зависимости устанавливаются внутри образа при сборке. После этого копируется остальной код приложения (`app.js`, `server.js` и при необходимости тесты). Задаются переменные окружения `NODE_ENV=production` и `PORT=8080`, порт 8080 объявляется через `EXPOSE`. Запуск выполняется командой `npm start` (соответствует `node server.js`). Исходный Dockerfile доступен в репозитории по пути `backend/Dockerfile`. Файл `.dockerignore` в каталоге backend исключает из контекста сборки каталог `node_modules` и служебные файлы, чтобы зависимости устанавливались только из `package.json` в рамках сборки образа.

Итог: серверная часть тестовой среды создана на Node.js и Express, реализует маршруты для статуса, проверки связи, заметок и документации API, использует PostgreSQL и Redis для проверки доступности сервисов и упакована в Docker-образ с установкой зависимостей в Dockerfile.
